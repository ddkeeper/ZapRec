# 极简录屏工具 (ZapRec) - 项目规格说明书

## 1. 项目概述

### 1.1 项目名称
**ZapRec**

### 1.2 项目类型
跨平台桌面录屏应用程序（优先适配 Windows，完美兼容 macOS。**注：因 Wayland/X11 显示服务器碎片化极其严重，为保障项目的极简与稳定，坚决不提供 Linux 官方支持**）。

### 1.3 核心定位
一款追求极致轻量、零等待导出的桌面录屏工具。剥离一切臃肿的后期编辑功能，专注于极简的**横向悬浮透明工具条**（全屏/窗口/区域/纯镜头）与双路音频捕获，实现录制即生成的丝滑体验。

### 1.4 目标用户
- 需要高频录制学术研究演示素材、日常 Vlog 记录、以及代码 Bug 复现的创作者与学生群体。
- 追求极致简洁界面，反感复杂参数配置的非专业用户。
- **对传统录屏软件“录完还要等进度条转码”极度反感的用户。**

### 1.5 运行环境要求 (Minimum System Requirements)
为保障 WebCodecs 硬件加速与流式写入的稳定性，设定以下基础门槛：
* **操作系统**：
  * **Windows**：Windows 10 (1809 及以上版本) 或 Windows 11。
  * **macOS**：macOS 11.0 (Big Sur) 及以上版本。
* **硬件要求**：
  * 处理器：64位 CPU (x64 或 ARM64 架构)。
  * 显卡：必须支持 H.264 硬件编码 (Intel QSV / NVIDIA NVENC / AMD AMF / Apple Silicon)。
  * 内存：最低 4GB RAM。

### 3.3 Electron 底层渲染与 GPU 调优 (Critical)
由于极度依赖硬件编码与透明窗口渲染，必须在 `main/index.ts` 中 `app.whenReady()` 之前注入以下 Chromium 底层参数，释放 GPU 性能：
* `app.commandLine.appendSwitch('enable-features', 'WebCodecsVideoEncoderHardwareAcceleration')`：强制开启 WebCodecs 硬件加速特性。
* `app.commandLine.appendSwitch('offscreen-use-shared-texture')`：优化 OffscreenCanvas 在区域录制裁剪时的 GPU 内存共享性能。
* `app.commandLine.appendSwitch('disable-gpu-sandbox')`：**(视环境备选)** 部分老旧 Windows 显卡驱动在沙盒模式下会导致 WebCodecs 崩溃，可作为回退诊断参数。

### 3.4 Instant Mode (即时模式) 底层数据流管线

为了实现 0 延迟导出，必须采用**纯内存流式管线架构 (Stream Pipeline)**。核心运转流程如下：

1. **环境准备 (Secure Context)**：WebCodecs 强依赖安全上下文，Electron 主进程必须在注册协议时声明 `privileges: { secure: true }`，确保渲染进程能正常唤起 API。
2. **音频混流 (Audio Mixing) 与双端差异化捕获**：
   * **Windows 路线**：利用 `desktopCapturer` 获取屏幕/窗口流时，直接挂载 `audio: { mandatory: { chromeMediaSource: 'desktop' } }` (Loopback 机制) 即可无损捕获系统内部声音。
   * **macOS 路线**：由于系统物理限制，必须检测并依赖第三方虚拟声卡驱动（如 BlackHole）。通过 `getUserMedia` 指定该虚拟音频设备 ID 来获取系统音。
   * **混音器 (Mixer)**：将捕获到的系统声音轨与麦克风声音轨接入 `AudioContext`，通过 `MediaStreamAudioDestinationNode` 实时合成为一条统一的立体声音频流，确保音轨严格对齐。
3. **硬件编码 (Encoding)**：
    * 视频流送入 `VideoEncoder`（配置 `hardwareAcceleration: 'prefer-hardware'` 优先调用 GPU），输出 H.264 数据块。
    * 混合音频流送入 `AudioEncoder`，输出 AAC 数据块。
4. **流式封装与 IPC 传输**：H.264 和 AAC 数据块持续喂给 `Mediabunny`，产出标准的 MP4 Chunks (`Uint8Array`)，通过 IPC 实时发送给主进程。
5. **瞬间落盘 (Disk I/O)**：主进程使用 `fs.createWriteStream` 持续将二进制块追加写入硬盘。结束录制时，前端触发停止信号，由 `Mediabunny` 补全文件尾部的 `moov box` 并调用 `stream.end()`。

## 4. 界面设计原型

### 4.1 主界面布局 (横向透明工具条模式)

采用**横向悬浮透明工具条 (Floating Toolbar)** 设计。主打极致的沉浸感与无干扰体验，配合毛玻璃特效完美融入系统桌面。

```text
╭──────────────────────────────────────────────────────────────────────────╮
│ ⣿ │ [🖥️ 全屏] [🪟 窗口] [⛶ 区域] [📷 纯镜头] │ [👤 摄像头] [🎙️ 麦克风] [🔊 系统音] │ ⚙️ │
╰──────────────────────────────────────────────────────────────────────────╯
```

*交互说明*：
1. **左侧拖拽把手**：`⣿` 区域为窗口拖拽区，允许自由放置在屏幕边缘。
2. **左中区 (核心触发器与分层逻辑)**：
   * **[🖥️ 全屏] (秒开)**：智能锁定当前所在屏幕，直接触发 3-2-1 倒计时开录。
   * **[🪟 窗口] (确权)**：必须先弹出应用窗口缩略图列表，选中目标后触发开录。
   * **[⛶ 区域] (交互闭环)**：点击唤出全屏透明遮罩进行鼠标拖拽框选。**框选后支持按下 `Enter` 键开录，或按下 `Esc` 键取消框选并返回主界面**。
   * **[📷 纯镜头]**：跳过桌面画面，录制全屏摄像头。(系统自动忽略并隐藏画中画)。
3. **右中区 (透明度状态预设)**：音视频开关。**开启 (ON)**：深色背景低透明度，图标锐利；**关闭 (OFF)**：暗色背景高透明度，融入背景。
4. **沉浸式收纳与托盘兜底**：录制时工具条隐藏。**托盘图标需支持右键呼出完整菜单（包含：停止录制、暂停/恢复、打开设置、退出程序）**，防止用户忘记快捷键。
5. **macOS 系统音防呆交互**：Mac 端的 `[🔊 系统音]` 默认处于**半透明置灰**状态。强制点击时弹窗提示引导安装虚拟声卡驱动（如 BlackHole），作为用户体验摩擦点 (Friction Point) 的柔性过渡。

## 6. 边缘情况与容错处理 (Edge Cases)

### 6.1 启动期：硬件与环境前置检测 (Pre-flight Checks)
应用启动后或用户点击录制前，必须完成以下静默检测（若不通过则阻断并告警）：
- [ ] **WebCodecs API 可用性**：检测 `typeof VideoEncoder !== 'undefined'` 且 `VideoEncoder.isConfigSupported()` 返回可行。
- [ ] **系统权限状态**：检测屏幕录制、麦克风、摄像头权限是否已授权（特别是 macOS）。
- [ ] **磁盘 I/O 探针**：向默认输出目录尝试写入一个 1KB 的 `.tmp` 隐藏文件并立刻删除，验证磁盘未满且具有写权限。

### 6.2 运行期：异常场景处理
| **异常场景** | **防御与处理机制** |
| :--- | :--- |
| **录制目标丢失 (窗口关闭/显示器断开)** | **核心容错**：监听底层流的 `onended` 事件。中途目标丢失时，自动触发 `stream.end()` 安全保存前半部分视频，并弹窗通知：“录制源已断开，视频已自动保存”。 |
| **磁盘空间耗尽 (ENOSPC)** | 主进程捕获 `ENOSPC` 错误，强制保存已录制片段，防止引发系统级雪崩。 |
| **断电或异常强杀进程** | 由于 MP4 强依赖尾部 `moov box`，断电会导致流文件永久损坏。此为即时写入架构的已知代价。 |
| **录制目标窗口被最小化** | **Electron 底层限制**：操作系统会停止渲染被最小化的窗口，导致视频流画面冻结（卡死在最后一帧）。**处理机制**：在底层通过 `BrowserWindow` 或原生模块监听目标窗口状态，一旦检测到被最小化，立即触发 `stream.end()` 安全截断视频并弹窗告警，防止生成损坏的死帧文件。 |

### 6.3 运维期：本地日志与诊断能力
作为无需联网的本地工具，必须具备完善的离线排障能力：
* **日志库选型**：引入 `electron-log`，接管主进程与渲染进程的 `console.log/error`。
* **日志落盘策略**：自动写入用户 AppData 目录（如 `%USERPROFILE%\AppData\Roaming\caplet\logs`），单文件最大 5MB，保留最近 3 份，防止占用空间。
* **一键导出诊断**：在“⚙️ 设置”面板中提供【导出诊断日志】按钮，一键打包硬件信息、Electron 报错栈和最近日志并打开所在文件夹，便于排查 WebCodecs 兼容性客诉。

### 6.4 录制中动态切换待优化问题 (Known Limitation)

| **问题描述** | **影响范围** | **优化方向** |
| :--- | :--- | :--- |
| **录制过程中无法动态控制辅助信息源** | 画中画 (PIP)、麦克风，系统声音在录制过程中切换状态时，未能实时同步到 RecordingEngine。 | 需要在 AudioMixer 中实现动态添加/移除轨道的接口，并在 UI 状态变更时触发引擎重新配置。具体实现方案：<br>1. **AudioMixer 增强**：添加 `enableTrack(trackName)` / `disableTrack(trackName)` 方法，支持运行时动态开关音频轨。<br>2. **RecordingEngine 增强**：暴露 `addAudioTrack()` / `removeAudioTrack()` 接口，支持在录制过程中增删音频流。<br>3. **UI 与引擎通信**：Toolbar 组件点击开关时，通过回调函数通知 App 层，由 App 层调用引擎接口完成动态切换。<br>4. **视频轨同理**：画中画 (PIP) 涉及视频轨叠加，需在 RecordingEngine 中实现多视频轨动态合成。 |
| **触发条件** | 录制过程中点击"画中画"、"麦克风"、"系统音"任一开关 | 点击开关时实时生效，无需等待停止录制 |

## 7. 构建配置与产物形态 (Build & Artifacts)

除了基础的 ASAR 保护和 macOS 权限声明（Entitlements）外，针对 `electron-builder` 的最终产物形态作出以下严谨规划：

### 7.1 Windows 产物形态 (x64 / arm64)
提供两种发行格式，覆盖不同用户偏好：
1. **NSIS 一键安装包 (`.exe`)**：默认推荐。配置 `oneClick: true`, `perMachine: false`。不显示繁琐的“下一步”，双击瞬间完成安装并启动，契合轻量级心智。
2. **便携版 (`.exe` Portable)**：针对极客用户和 U 盘场景，提供无需安装的 Portable 版本（即开即用，配置随身生成）。

### 7.2 macOS 产物形态 (Universal)
* **目标架构**：配置 `mac.target: "dmg"`, `mac.arch: "universal"`。将 Intel (x64) 和 Apple Silicon (arm64) 打包为单一的 Universal Binary (通用二进制文件)，避免小白用户下错版本导致性能折损。
* **签名与公证**：强制要求 Apple Developer 证书签名与 Notarization，否则将被 macOS 拦截报错。

* 针对 macOS 进一步优化 Apple Silicon (M系列芯片) 的原生硬件编码调度。

### 待完成问题
**现象记录**：在“录制指定窗口”模式下（非全屏/非区域），由于操作系统底层（Windows/macOS）的窗口捕获机制，覆盖在目标窗口上方的 PIP 悬浮窗可能会被系统自动剔除，导致录制的 MP4 中丢失 PIP 画面。
**当前决策**：作为纯物理录制方案的原生限制，该现象属于系统级物理现实。当前版本暂不处理此边界情况，作为已知特性记录在案，以保持核心架构的极简与稳定。
